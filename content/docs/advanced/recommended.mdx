Claude Codeをもっと便利にするプラグインとスキルを紹介する。コミュニティで人気のSkillはコピーしてそのまま使えるので、`~/.claude/skills/` に置くだけでOK。

## 公式プラグイン：Superpowers

Claude Code開発チームが提供する公式プラグイン。インストール方法は[Pluginsのインストール](/docs/plugins/installing-plugins)を参照。

| スラッシュコマンド | 内容 |
|---|---|
| `/commit` | 変更内容を解析してConventional Commits形式のコミットを自動生成 |
| `/review` | コードレビューを実行。改善点・バグ・セキュリティリスクを指摘 |
| `/debug` | 体系的なデバッグ手順でバグの根本原因を特定 |
| `/brainstorming` | 実装前に要件・設計・制約を対話形式で整理 |
| `/writing-plans` | ブレインストーミング後に詳細な実装計画を作成 |
| `/executing-plans` | 作成した実装計画を段階的に実行 |
| `/tdd` | テスト駆動開発の手順でコードを実装 |
| `/finishing-a-development-branch` | 実装完了後のPR作成・マージ手順をガイド |
| `/requesting-code-review` | レビュー依頼文の作成とチェックリストの確認 |
| `/systematic-debugging` | 症状→原因→仮説→検証の順で体系的にデバッグ |

## コミュニティ人気Skill集

> コピーして `~/.claude/skills/<name>.md` に保存すれば即使える。

### Git・バージョン管理

#### `changelog` — CHANGELOGを自動生成

```markdown
---
name: changelog
description: git logからCHANGELOG.mdを自動生成する
---

git log --oneline でコミット履歴を取得し、
feat/fix/docs/chore などのタイプ別に整理して
CHANGELOG.md にKeep a Changelog形式で追記する。
今日の日付でバージョンセクションを作成すること。
```

#### `squash` — コミットをまとめる

```markdown
---
name: squash
description: 作業中の細かいコミットを1つにまとめる
---

現在のブランチでmainから分岐したコミット一覧を表示し、
ユーザーに確認を取ってからインタラクティブrebaseで
1つのコミットにまとめる。コミットメッセージは変更内容を
要約して自動生成する。
```

#### `undo` — 直前の変更を安全に元に戻す

```markdown
---
name: undo
description: 直前のコミットや変更を安全に取り消す
---

git statusとgit logで現在の状態を確認し、
ユーザーに「何を元に戻すか」を確認してから実行する。
コミット取り消し・ファイル単位の復元・ステージング解除を
状況に応じて使い分ける。絶対にgit push --forceは使わない。
```

### コードレビュー・品質

#### `security-check` — セキュリティ診断

```markdown
---
name: security-check
description: コードのセキュリティリスクをチェックする
---

プロジェクト全体を走査して以下を確認する：
1. ハードコードされたAPIキー・パスワード・トークン
2. SQLインジェクションの可能性がある箇所
3. XSSの可能性がある箇所（未エスケープの出力）
4. 認証・認可の抜け漏れ
5. 依存パッケージの既知の脆弱性（npm audit / pip audit）
問題をリスト形式で報告し、深刻度（高・中・低）を付ける。
```

#### `complexity` — 複雑なコードを検出

```markdown
---
name: complexity
description: 複雑すぎる関数・ファイルを見つけて報告する
---

以下の観点でコードを分析する：
- 100行を超える関数
- ネストが3段以上の条件分岐
- 引数が5つ以上の関数
- 重複しているコードブロック
各問題箇所をファイル名・行番号付きで報告し、
リファクタリングの提案も添える。
```

#### `type-check` — 型安全性の確認

```markdown
---
name: type-check
description: TypeScriptの型エラーと any の使用箇所を報告する
---

tsc --noEmit を実行して型エラーを一覧表示する。
あわせてコード中の any 型の使用箇所を検索し、
より適切な型への置き換え案を提示する。
```

### ドキュメント・コメント

#### `jsdoc` — JSDoc コメントを生成

```markdown
---
name: jsdoc
description: 関数にJSDoc形式のコメントを追加する
---

指定されたファイル内のすべての関数・メソッドに
JSDoc形式のコメントを追加する。
@param・@returns・@throws・@example を含める。
既存のコメントは上書きしない。
```

#### `readme` — READMEを生成・更新

```markdown
---
name: readme
description: プロジェクトのREADMEを生成または更新する
---

package.json・ディレクトリ構造・主要ファイルを読んで
以下のセクションを含むREADME.mdを生成する：
- プロジェクト概要（1〜2行）
- 機能一覧
- インストール手順
- 使い方（コード例付き）
- 環境変数の説明
- ライセンス
既存のREADMEがある場合は差分だけ更新する。
```

#### `api-docs` — APIドキュメントを生成

```markdown
---
name: api-docs
description: APIエンドポイントのドキュメントを自動生成する
---

ルーティングファイルを走査してすべてのAPIエンドポイントを特定し、
docs/api.md にOpenAPI風のドキュメントを生成する。
各エンドポイントについて：メソッド・パス・リクエスト形式・
レスポンス形式・エラーケースを記載する。
```

### テスト

#### `test-coverage` — テストが足りない箇所を探す

```markdown
---
name: test-coverage
description: テストが書かれていない関数・コンポーネントを報告する
---

srcディレクトリを走査してすべての関数・コンポーネントを特定し、
対応するテストファイルに同名のテストがあるか確認する。
テストが存在しない箇所をリストアップし、
テストの書き方の雛形も一緒に提示する。
```

#### `e2e-plan` — E2Eテストのシナリオを設計

```markdown
---
name: e2e-plan
description: E2Eテストシナリオの一覧を設計する
---

アプリの主要な画面・機能を確認し、
ユーザー視点でのE2Eテストシナリオを設計する。
ハッピーパスと異常系（エラー・権限なし・タイムアウト）を含め、
Playwright / Cypress で実装しやすい形式で出力する。
```

### 作業効率化

#### `standup` — 朝会報告文を自動生成

```markdown
---
name: standup
description: 昨日の作業をgit logから読んでスタンドアップ報告文を生成する
---

git log --since="yesterday" でコミットを取得し、
以下の形式でスタンドアップ報告文を生成する：
- 昨日やったこと
- 今日やること（直近のIssue・TODOから推測）
- ブロッカー（未解決のエラーがあれば記載）
Slack投稿用に短くまとめること。
```

#### `todo-scan` — TODOコメントを一覧化

```markdown
---
name: todo-scan
description: コード中のTODO・FIXME・HACKを一覧化する
---

プロジェクト全体から TODO・FIXME・HACK・XXX コメントを
検索してファイル名・行番号・内容をリスト化する。
古そうなものや重要度が高そうなものにフラグを立てる。
```

#### `env-check` — 環境変数の整合性確認

```markdown
---
name: env-check
description: .envと.env.exampleの整合性をチェックする
---

.env.example に記載された変数が .env にすべて存在するか確認する。
.env にあって .env.example にない変数（漏れている可能性）も報告する。
機密値は表示せず、キー名だけ表示する。
```

#### `migrate` — DBマイグレーションを安全に実行

```markdown
---
name: migrate
description: データベースマイグレーションを安全な手順で実行する
---

以下の手順でマイグレーションを実行する：
1. 現在のDBスキーマとマイグレーション一覧を確認
2. 未適用のマイグレーションを表示してユーザーに確認
3. バックアップを取るよう促す
4. マイグレーションを実行
5. 結果を報告し、ロールバック手順を案内する
```

### フロントエンド

#### `a11y` — アクセシビリティチェック

```markdown
---
name: a11y
description: UIコンポーネントのアクセシビリティ問題を指摘する
---

指定されたコンポーネントまたはディレクトリを走査して
以下のアクセシビリティ問題を確認する：
- imgタグにaltがない
- ボタンにラベルがない（aria-label等）
- フォームのlabelとinputの紐付けがない
- キーボード操作できないインタラクティブ要素
- カラーコントラストが低い可能性のある組み合わせ
各問題にWCAG 2.1の達成基準番号を付けて報告する。
```

#### `storybook` — Storybookのストーリーを生成

```markdown
---
name: storybook
description: ReactコンポーネントのStorybookストーリーをCSF3形式で生成する
---

指定されたコンポーネントのprops定義を読んで
CSF3形式のStorybookファイルを生成する。
Default・WithProps・EdgeCases の3ストーリーを最低限含め、
コンポーネントと同じディレクトリに配置する。
```

### チーム・プロジェクト管理

#### `issue` — GitHubイシューを作成する

```markdown
---
name: issue
description: 発見したバグや改善点をGitHubイシューとして起票する
---

ユーザーからバグ・改善点の説明を聞いて、
gh issue create コマンドで以下の形式のイシューを作成する：
- タイトル（簡潔に）
- 再現手順
- 期待する動作
- 実際の動作
- 環境情報
ラベル（bug/enhancement）も自動で付与する。
```

#### `pr-review-checklist` — PRレビューチェックリスト

```markdown
---
name: pr-review-checklist
description: PRレビュー時の確認チェックリストを出力する
---

以下のチェックリストを表示してレビューをサポートする：
- [ ] 変更の意図がPR説明文から読み取れるか
- [ ] テストが追加・更新されているか
- [ ] 破壊的変更がないか（あれば明記されているか）
- [ ] セキュリティ上の問題がないか
- [ ] パフォーマンスへの影響はないか
- [ ] ドキュメントが更新されているか
各項目を確認しながらコードを読み、コメントを生成する。
```

#### `onboarding` — 新メンバー向けプロジェクト説明

```markdown
---
name: onboarding
description: 新しいメンバーにプロジェクトの全体像を説明する
---

以下を読んでプロジェクトの全体像を説明するドキュメントを生成する：
- README.md
- package.json / requirements.txt
- ディレクトリ構造
- 主要なファイルの役割
出力形式：ONBOARDING.md として保存し、
「環境構築」「コードの読み方」「開発フロー」の3セクションにまとめる。
```

## コミュニティSkillの探し方

GitHubで以下のキーワードで検索するとコミュニティ作成のSkillが見つかる：

- `claude-code skills`
- `claude-code plugin`
- `claude code slash command`

自分のSkillを公開するときは同じキーワードをトピックに設定しよう。

## Skillを作るときのコツ

| コツ | 説明 |
|---|---|
| 動詞で始める | 「〇〇を作る」「〇〇を確認する」と具体的に |
| 出力形式を指定する | 「箇条書きで」「マークダウンで」など |
| スコープを絞る | 1つのSkillには1つの責務だけ |
| 確認ステップを入れる | 破壊的操作の前は必ずユーザー確認 |
| まずSkillから | 慣れてきたらPluginにまとめる |
